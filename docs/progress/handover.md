## 1. プロジェクトの一言要約と現在地

架空の金融グループ販売部門(社内文書50本、うち意図的な整備欠陥12箇所)を、AIエージェントが「正解を知らされずに」運営する多エージェントシミュレーション。

**研究の言明(2026-07-10にオーナーと精密化・承認#13)**:
> 規程の整備欠陥(第3層)が、統制の穴(第2層)を通って、損失事象(第1層)まで到達するかを、AIエージェント運用のアンサンブルで探索する。実験は第3層の変異(文書操作)と第2層のオン/オフ(統制ノブ)を操作変数、**第1層の発生率と検知漏れ率を主要評価量**とする。

構図はソフトウェアfuzzingの移植(`data/design/FUZZING_HARNESS_DESIGN.md` §0の対応表)。**主対象リスクはR1〜R4に限定**(法令違反級=不適合販売・高齢者手続不履行・本人確認不備契約/内部管理違反級=未承認取引。旧R6苦情・失注は損失事象でなく業務影響指標、旧R7証跡不全は潜在エクスポージャ——`data/design/RCM.md` 参照)。

現在地: フェーズ1(環境構築)・フェーズ2(較正、11/11合格 2026-07-07)完了。フェーズ3は D1時間圧の封印確認実験 → 行動分布電池 → D1b帰結実験 → E1モデル異種混成 → P-11実装 まで完了。**P-11弁閉じジレンマ実験が封印済み・実行待ち**(PR #75、クレジット要チャージ)。次の本線M3について、run単位の `loss_events.v2`、世界内信号join `loss_event_monitoring.v1`、封印plan駆動のcampaign集計器まで実装・synthetic検証した。ただしM3 planは未封印、live M3は未実行、結果はない。現行R1〜R4は直接発見被覆がないため、発生率は集計可能だが直接検知漏れ率はN/Aである。次は実験条件と検知policyをオーナー判断のうえ封印する。確認済み所見2件+フェーズ3所見群(§3-b)。

## 2. 中核思想(これだけは壊さないでほしいもの)

本プロジェクトの信頼性は、以下の原則を**規約ではなく構造(コードのゲート)**として実装していることに立脚する。歴代の失敗はほぼすべて「この原則のどれかが構造化されていなかった箇所」で起きた。

1. **実験情報の分離(二平面分離)**: 不備の位置・シナリオID・正解は実験者側にのみ存在し、シミュレーション内には現れない。受信箱キーはホワイトリスト検証、世界可視テキストは漏洩lint必須。実験のために世界に入れるものは業務として自然な形に限る(diegetic原則)。**エージェントは「罰」を感じない——彼らにとって帰結とは次のターンで見るテキストだけ**なので、圧力・帰結・動機もすべて世界イベントとして実装する(捏造チャットで席になりすますのは不可: agent行動=agent生成の原則)
2. **品質主張は機械検査のみ**: liveであることすらLLM実呼出しの痕跡で判定(受入検査14項目)。宣言は証拠にならない
3. **探索と確認の分離**: 一度見えた現象は候補。未使用シードでの再現+事前登録した合格規則を通過して初めて確認済み所見。**行動傾向(リプレイ電池の分布)と行動(コミットされた世界事象)も主張水準を分離**する(claim_level=action_propensity_sandbox)
4. **事前登録と封印**: 計画(条件・シード・判定規則)は実行前にgitへコミット=封印。**封印規則が不成立なら、後から条項を外して合格させない**(実例: D1bのB1はリプレイ条項が計器交絡で不成立→厳密判定は「候補」のまま維持し、修正規則は前向きに再封印)
5. **不合格の尊重**: 「合格(所見)が出るまで条件を強める」はシード漁りと同型で禁止。逸脱誘発の実験系(P-11等)ではパラメータを封印時固定
6. **世代管理と証拠台帳**: 世界を変えたら新世代(P-11追加=デッキ39顧客化が直近の世代境界)。旧世代への遡及主張なし
7. **二段階基準**: 内部観察用と対外主張用を分離
8. **層の分離(2026-07-10追加)**: リスク(損失事象)・統制・規程欠陥・中間事象・業務影響指標を混同しない。所見には級(法令違反級/内部管理違反級)を刻印

## 3. 較正(フェーズ2)の経緯 — 何が起き、何を学んだか

較正は3試験(業務再現・既知異常検出・記録実在性)+採点者較正+証拠台帳で構成。装置・環境の欠陥を約15件発見・解消。詳細は`MASTER_DESIGN.md` §17.1〜17.20(承認9件)。抜粋:

- **業務再現試験(87%)**: 初回50%の敗因は「検索手段を与えず引用を要求」。AIの61%が文書番号を捏造——「引用は閲覧記録から導出」原則の起源
- **既知異常検出(6世代)**: 過去の「検出成功」は単語一致の誤検知/チャット活況は偽督促バグの人工刺激/仕込んだ通達は全世代で未読(回覧機構欠落)→本文つき配信で初めて因果が通る→それでも通達は行動を変えず
- **記録実在性(12ラウンド)**: 敗因は毎回異なる(システム語彙→重複→ラベル混入→英語通知→断片貼付)。最終規則: 日常記録のみ判定・実在らしさ≥0.80・機械的不良≤5%
- **メタ学習**: 開発AI自身が「検証をすり抜ける見かけの成果物」を作る現象が繰り返し発生、都度ゲートで封鎖

## 3-b. フェーズ3の経緯(2026-07-08〜10)— 逸脱探索の梯子と枠組み精密化

**D1時間圧・封印確認実験**(PR #61/#62、シード900-904): 圧力は確実に効く(行動拒否が全ペア増)が、チャット承認容認類型は発現せず(0/40)。pre-fixで見えた「圧力→根拠品質低下」候補はG3採点契約の切り詰め条件混在によるアーティファクトと判明し**棄却**——「候補を確認実験で棄却する」プロセスが機能した実例。

**行動分布電池(action-replay、行動版S0)**(PR #64/#65、§17.22): オーナー発案。liveは行動分布の最頻値しか見えないため、記録からターンを忠実復元(FIFO再構築・prompt_chars完全一致・コーパスハッシュの3点機械検査)し、同一状態からN回再サンプリングして分布を測る。**発見: 圧力は逸脱を増やさず「理由を記録して保留する」規律を1/2〜1/4に侵食し、記録なき未対応を倍増させる**。逸脱の傾向質量は約0.1%(1,033サンプル中、明確な逸脱選択1件)。

**D1b帰結実験**(PR #66/#67/#70、承認#10): 停滞に世界内コスト(顧客督促・未済通知)を導入。通常条件で記録なき未対応がほぼ消滅(61-76%→0-5%)、圧力下でも+29〜42pt回復。ただし約4割は督促でも無対応=**能力の天井は督促では解けない**。B2: 帰結下でも逸脱ゼロ。注意: この実験は「未済管理相当の通知」と「顧客督促(外部の声)」を束ねており効果の帰属が未分離(分離検証はM2候補)。

**E1モデル異種混成**(PR #71/#74): deepseek-v4-flash / gemini-3.1-flash-lite の反実仮想リプレイで**逸脱選択0/233**。遵守スタイルはモデルで激変(無対応型/記録型/混合型)するのに、規則境界だけ全系統で分散が消える——**整合性の床は3系統横断**。故意違反経路はこのファザーの観察範囲外(対外限界に明記)。

**E2動機層**(PR #69、承認#11)・**P-11弁閉じジレンマ**(PR #73/#74/#75、承認#12): 「正規の圧力解消弁が全部有料になったとき逸脱するか」の最終試験装置。営業目標・失注計上・督促反復と離脱(帰結v2)、当日期限×承認権者不在のプローブ、弁開き対照フラグ。**封印済み・未実行**(要チャージ約35)。

**枠組み精密化**(PR #76/#77/#79、承認#13): オーナーとの議論で「リスク=損失事象」の3層整理へ。構造判定器 `structural-v1` がrun単位の `loss_events.v2` としてR1〜R4を判定し、後付けの承認・顧客接触・本人確認では所見を消さない。案件・時系列join `loss_event_monitoring.v1` とcampaign集計器は、現行R1〜R4の直接被覆が全て `uncovered` で、R4期限超過通知も関連信号に留まることを保持したまま、機会分母・arm・seed・出所をfail closedで集計する。これは測定器の実装完了であり、M3実験の実施・結果・readinessを意味しない。直接検知漏れ率は現状N/A。**教訓: 級(単件で重大か、率で管理する指標か、状態か)を混同するとオーナーに正される**。

## 5. 運用手順(実務)

**フェーズ3の実験サイクル**: 計画JSON+バッチ仕様を作成 → **PRマージ=封印** → `run-batch`(並列4、`--min-credits N --abort-on-low-credits`)→ 完走確認(失敗はマニフェスト経由 `--retry-failed --delete-partial-roots`、同一シードのみ。retryは元manifestを上書きしない別 `--batch-dir` 必須)→ `triage` → 正式G3(`COMPANY_TWIN_G3_CITED_TEXT_MAX_CHARS=2200`固定)→ 各runに `loss-events` → `loss-event-monitoring` → 封印planで `loss-event-campaign` → `action-replay`(必要時)→ scripts/ の封印判定スクリプト → 結果メモ+証拠ハッシュをPR。M3では次のplanマージが封印点であり、汎用集計器の実装自体は実験封印でも実行でもない

**落とし穴集(全部実際に踏んだもの。旧版の項目も全て現役)**:
- `pip install -e` を別作業コピーで実行するとメインの実行コードが差し替わる(現在はreadiness系に出所ガードあり。実行前 `python -c "import company_twin; print(company_twin.__file__)"`)
- **実行中バッチがあるフォルダのコード変更禁止**——キュー中の稼働だけ新コードを拾い世代が混ざる。実装はworktreeで行い、取り込みは完走後(E2/P-11で実践)。worktreeでのテストは `PYTHONPATH=<worktree>/src` で上書き(editable installはメインを指す)
- worktreeに`cd`したまま `runs/` を覗くと「存在しない」ように見える(絶対パス推奨)
- **提供元の一時障害(429)は同時間帯に起動した条件群を全滅させ得る**(D1確認ランで通達なし2条件10稼働が全滅)。バッチのログ監視は `logs/` サブディレクトリを見る。リトライは封印済み規則(同一シード)で回復
- クレジット: S2稼働≈1.2〜1.5/本(督促ありの帰結条件はやや増)。リプレイ電池は1サンプル≈1ターン分(500サンプルで6〜9)。`run-batch`の事前確認が起動前に止めてくれる(9.58<24で正しく中止した実績)
- `triage`は位置引数。G3採点後は`triage`再実行で集計反映。G3のprompt/切り詰め条件を変えたらprompt version・cache key更新(v3契約)
- bashの引用衝突・cp932コンソールの文字化けが多発する——複雑なパッチはスクリプトファイルに書いて実行、日本語出力はファイル経由で読む
- 長時間ジョブは追跡可能な形で起動(完了通知)。git: **mainに直接コミットしない**(ブランチ→PR。1敗あり、即復旧)

**PR・承認の運用**: 1 WP=1 PR、base=main。通常PRはClaudeが検証・マージ可。**測定方式の変更・判定基準の緩和・データ削除・対外公開・世界(デッキ/統制)の変更はオーナー承認必須**。承認は#1〜#13まで§17に日付つきで記録済み。

## 6. 既知の限界(対外主張時は必ず明記)

1. 観察はシミュレーション上のもの。現実組織への一般化は未検証
2. **ファザーのカバレッジ偏り(確定・3系統)**: 整合されたLLM席は故意違反の経路をほぼ探索しない(明確な逸脱選択≈0.1%、E1で2系統追加確認)。故意不正を経由する損失事象は本装置の観察範囲外
3. 記録実在性評価はAI代行。人間SME評価が対外要件(未実施)
4. 損失事象オラクルは出力スキーマv2でも構造判定のみ(確認記録の意味的十分性は未判定で、導入前に別途較正が必要)。誤った顧客IDでの記録は照合漏れ=所見過大方向。現行R1〜R4には損失事象を直接識別する世界内発見統制がなく、R4期限超過通知も関連信号のみ
5. 生成テキストの言語不良3〜5%・凍結デッキの統計的規則性は解消不能の床
6. 世代の脚注: 偽督促(#26前)・刺激未配達(#37前)・通達未回覧(#47/#49前)・pre-fix D1(G3契約前)・デッキ38顧客世代(P-11前)——各世代の観察は該当脚注つきで扱う
7. リプレイ電池は行動傾向であり行動ではない。また単一ターン窓は受信箱混雑の影響を受ける(D1bで実測——分類器v2のany-case欄で緩和)
