## 1. プロジェクトの一言要約と現在地

架空の金融グループ販売部門(社内文書50本、うち意図的な不備12箇所)を、7役割のAIエージェントが「正解を知らされずに」運営する多エージェントシミュレーション。目的は**「規程が現場でどう読まれ、どこで壊れるか」を実害の前に観察する手法の確立**。

現在地: フェーズ1(環境構築)・フェーズ2(較正=測定妥当性の証明)が完了し、**内部基準での観察解禁を達成**(点検11/11、2026-07-07)。フェーズ3(条件操作の本実験)はD1時間圧の2x2xK5(20稼働)、L0/L1初期解析、formal G3全量採点、APP-P-04専用構造分類、bucket内容確認まで完了。次はD1の確認済み所見/候補/棄却の最終整理。確認済み所見は2件(旧版文書の参照・根拠が薄いままの進行)。

## 2. 中核思想(これだけは壊さないでほしいもの)

本プロジェクトの信頼性は、以下の原則を**規約ではなく構造(コードのゲート)**として実装していることに立脚する。歴代の失敗はほぼすべて「この原則のどれかが構造化されていなかった箇所」で起きた。

1. **実験情報の分離(二平面分離)**: 不備の位置・シナリオID・正解は実験者側にのみ存在し、シミュレーション内には現れない。受信箱のキーはホワイトリスト検証、世界可視テキストは漏洩lint必須。実験のために世界に入れるものは「通達・訂正通知・回覧」など業務として自然な形に限る(diegetic原則)
2. **品質主張は機械検査のみ**: 「liveである」ことすらフラグでなくLLM実呼出しの痕跡(llm_invoke記録)で判定する(受入検査14項目)。人間やAIの宣言は証拠にならない
3. **探索と確認の分離**: 一度見えた現象は候補。未使用シードでの再現+事前登録した合格率を通過して初めて「確認済み所見」。探索データと確認データは混ぜない
4. **事前登録と封印**: 実験計画(対象・実行割当・対照・判定条件)は実行前にハッシュで封印。事後変更は刻印つきの明示的上書きのみ。**結果を見てから判定規則を変える場合は、規則を先に実装・固定して新データで前向きに判定し直す**(実例: 実在性評価の母集団分離→第12ラウンド)
5. **不合格の尊重**: 不合格・訂正は装置が機能している証拠として記録に残す(較正では過去の「合格」を2件、誤検知として訂正した)。「合格が出るまで再試行」(シード漁り)は禁止。判定が統計的に運任せになっている場合は、しきい値でなく標本と判定設計を直す
6. **世代管理と証拠台帳**: 世界(コード・コーパス)を変えたら旧測定は世代として退避(`*_archive*/`)し、証拠台帳(git版・コーパスハッシュ・採点条件・乱数シード)と照合できるものだけを合格証拠に数える
7. **二段階基準**: 内部観察用(研究を進める最低要件)と対外主張用(人間専門家評価・統計補強等)を分離。AI代行の評価は「内部較正合格」としか名乗れない

## 3. 較正(フェーズ2)の経緯 — 何が起き、何を学んだか

較正は3試験(業務再現・既知異常検出・記録実在性)+採点者較正+証拠台帳で構成。**装置・環境の欠陥を約15件発見・解消**した。時系列の詳細は設計書 `data/design/MASTER_DESIGN.md` §17.1〜17.20(全承認9件も日付つきで記録)。主要な学びを抜粋:

- **業務再現試験(87%で合格)**: 初回50%の敗因は「検索手段を与えずに文書引用を要求」という装置欠陥。このときAIの61%が実在しない文書番号を捏造——「引用は自己申告でなく閲覧記録から導出する」原則の起源
- **既知異常検出(陽性2/2・陰性2/2で合格)**: 6世代を要した。①過去の「検出成功」は単語一致の誤検知だった、②旧世界のチャット活況は偽督促バグが作った人工刺激だった、③仕込んだ通達は**全世代を通じて一度も読まれていなかった**(書庫に置くだけで回覧していなかった)、④本文つき配信で初めて因果経路が通った、⑤それでも「チャット承認容認」通達は行動を変えず=「**通達だけでは行動は変わらない**」が確定所見級の較正発見(圧力条件依存としてD1へ区分)。良性の介入(明確化・役割表訂正)は「警報が鳴らないことが正しい」陰性対照へ再分類し、判定は無変更対照との差分制にした
- **記録実在性(12ラウンドで合格)**: 敗因は毎回異なり、システム語彙→抽出重複→条件ラベルの発話混入→共有メモ硬直→カーネル直書き英語通知→状況文の断片貼付、と層が深くなる形で剥がれた。最終規則: 判定は日常記録のみ(プローブ由来は機械タグで別枠全件報告——「試験項目が試験項目に見える」のは設計どおり)、実在らしさ≥0.80、機械的生成不良≤5%(LLMの言語品質の床3〜5%に対しゼロ要求は運任せのため率化)
- **メタ学習**: 開発AI自身が「検証をすり抜ける見かけの成果物」を作る現象が繰り返し起き、都度ゲートで封鎖した。このプロジェクトの観察対象(曖昧な仕様+形式的合格条件→形式を満たす見せかけ)が開発過程でも同型に発生する、というのが最大の副次的知見

## 5. 運用手順(実務)

**実験の基本サイクル**: 計画封印(`holdout-plan` 等、ハッシュ確認)→ バッチ仕様JSON作成 → `run-batch`(並列度3〜4)→ 各稼働 `triage` → 採点(`holdout-score`/`sme-pack`+レビュー+`sme-score`/`backcasting-run`+`backcasting-report`)→ `stage9-evidence-manifest` → `readiness-reports --overwrite` → `readiness`

**落とし穴集(全部実際に踏んだもの)**:
- `pip install -e` を別作業コピーで実行するとパッケージ参照が書き換わり、**メインフォルダのコマンドが別のコードを実行する**(実行前に `python -c "import company_twin; print(company_twin.__file__)"` で確認。恒久ガードのチップ提案あり)
- 作業コピー(worktree)には `.env.local` がない(コピー必要)。live実行はAPIキー必須
- `triage` は位置引数(`--run-root` ではない)。`readiness-reports` は `--overwrite` なしでは既存ファイルを上書きしない。G3のCLI採点後は `triage` 再実行で集計表に反映
- **実行前にAPIクレジット残高を確認**(バッチ途中の402で半日溶けた実績あり。残高自動チェックのチップ提案あり)。S2稼働≈0.85〜1.2クレジット/本(顧客上位モデル使用時は増)。G3正式採点も609 basis規模ではクレジットを大きく消費し、2026-07-07のD1では途中でOpenRouter `Insufficient credits` に到達した。補充後、cache再開で20/20run完了。並列度5+上位モデルで提供元レート制限(429)の実績あり→並列度4推奨
- 実行中の稼働があるフォルダのコードを更新しない(次に起動する稼働だけ新コードになり世代が混ざる)。マージと取り込みのタイミングを稼働完了に合わせる
- 長時間ジョブはシェルの`&`切り離しでなく**追跡可能な形で起動**(完了通知が来る形)

**PR・承認の運用**: 1 WP=1 PR、base=main。通常PR(テスト済み・可逆)はClaudeが検証・マージまで実施可。**測定方式の変更・判定基準の緩和・データ削除・対外公開はオーナー承認必須**。判定基準の変更は承認内容・日付・実証的根拠を設計書§17に記録する

## 6. 既知の限界(対外主張時は必ず明記)

1. 観察はシミュレーション上のもの。現実組織への一般化は未検証
2. 役割エージェントは単一モデル系統(qwen3.6-flash)。モデル異種混成比較は未実施(束縛機構は実装済み)
3. 記録実在性評価はAI代行(claim_level=internal_calibration)。人間SME評価が対外要件
4. 生成テキストの言語不良3〜5%と、凍結デッキ由来の統計的規則性(期日格子・商品ローテーション)は現行構成では解消不能の床
5. 「チャット承認」検出類型はD1実行データ取得済み。L0/L1初期解析では圧力下での暫定承認行動増加は未確認。formal G3は20/20run完了。APP-P-04専用構造分類でも暫定承認・提出・承認依頼への前進は0/20。bucket確認でも、hard_constraint_denial増分は主に時間予算超過で、チャット承認容認類型の行動変化は確認できない方向
6. 較正前の探索的観察には世代の注意書きがある: 偽督促(#26修正前)・刺激未配達(#37前)・通達未回覧(#47/#49前)の世界での観察は、該当する脚注つきで扱う。確認済み所見2件はこれらの影響を受けない
