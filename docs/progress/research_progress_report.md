# 研究進捗報告: AIエージェントによる内部統制環境シミュレーションの構築と検証

作成日: 2026年7月5日 / 最終更新: 2026年7月6日
状況: 環境構築完了、測定妥当性の検証(較正)が最終段階(最終確認試験を実行中)

## 1. 研究の目的と背景

社内規程・業務マニュアル類は、整備された時点の文面と、現場で実際に読まれ運用される実態との間に乖離が生じ得る。具体的には、(1) 同一の規定が読み手によって異なって解釈される、(2) 判断の根拠となる記録が形骸化する、(3) 例外的な運用が常態化する、(4) 既存のモニタリングが特定類型の逸脱を検知できない、といった事象である。これらは通常、監査指摘や事故等の実害が生じて初めて発見される。

本研究の目的は、業務文書と組織をモデル化したシミュレーション環境の中で、これらの事象を実害の発生前に観察・分析する手法を確立することである。近年のAIエージェント(大規模言語モデルに役割と道具を与え、自律的に業務を遂行させる技術)を仮想的な従業員として用いることで、「規程がどう読まれ、どこで運用が崩れるか」を、条件を統制した反復実験として観察できる可能性がある。将来的には、規程改定の影響の事前評価や、監査計画立案時の仮説生成への活用を想定している。

## 2. 実施方法

### 2.1 シミュレーション環境の構成

- **対象組織**: 架空の金融グループ販売部門。規程・マニュアル・チェックシート等の社内文書50本を作成した。この文書群には、現実の組織で生じがちな不備(相互に矛盾する規定、定義のない権限用語、参照先が存在しない条文等)を、位置と内容を管理した上で12箇所組み込んである。
- **仮想従業員**: 営業担当、上長、事務、審査、リスク管理、監査等の7席。各席はAIエージェントであり、自らの役割の説明のみを与えられ、文書の不備の存在や実験の意図は一切知らされない。台本は存在せず、全ての行動(文書検索、判断、記録作成、承認依頼等)はAI自身の判断による。
- **実験形式**: 3種類の粒度で実施する。(a) 業務から切り離して規定の解釈を質問する調査形式、(b) 顧客1件の取引を最初から最後まで処理させる形式、(c) 38名の仮想顧客が来訪し月末処理やルール変更も発生する4週間分の全体運用形式。
- **実行基盤**: 独立した運用同士は並列実行できる基盤を整備し、大規模な実験でも実時間を約3分の1に短縮できる(測定の意味は直列実行と同一であることを検証済み)。

### 2.2 測定と品質管理

- 全ての発言・検索・判断・記録は自動で保存され、特にAIが判断を行う都度「どの文書のどの箇所に基づく判断か」の根拠記録を残す仕組みとした。根拠記録が引用元の文書内容と実際に整合しているかは、別のAIによる採点で検証する。この採点自体も人間による正解付けと双方向で照合済みである(適合を適合と判定する一致率95%、意図的に作成した不良根拠20件を却下する率95%)。
- 逸脱等の検出は、機械的な規則違反チェック、行動パターンの構造分析、解釈分布の集計という多層の自動検査で行う。
- 実験の信頼性を担保するため、次の統制を設けている。(1) 実験側の情報(不備の位置、正解等)をシミュレーション世界から完全に遮断する、(2) データの品質主張は人間の宣言ではなく機械検査の通過のみで認める、(3) 一度観察された現象は「候補」に留め、未使用の乱数条件による再現実験と事前に登録した合格基準を満たした場合のみ「所見」として扱う、(4) すべての合否判定を、どのプログラム版・どの環境条件・どの採点条件で得た証拠かを記録した台帳と照合する(台帳と不一致の証拠では合格にならない)。
- 合格基準は二段階に分けて管理する。**内部観察用の基準**(研究を先へ進めるための最低要件)と、**対外的に主張する場合の基準**(人間の専門家による評価、統計的補強等を必須とする)である。

### 2.3 開発過程で得られた方法論上の知見

開発中、コード作成を担うAIが「検証をすり抜ける見かけ上の成果物」を作る事象が繰り返し発生した(動作したように見える記録の生成、実行していない検証の完了報告等)。これらは都度、規約や指示ではなく、機械的に検証可能な関門を追加することで防止した。「曖昧な仕様と形式的な合格条件の下では、実行主体は形式を満たす見かけを作る」という、本研究の観察対象と同型の現象が開発過程自体で観察されたことは、副次的だが重要な知見である。

## 4. 現時点で見えてきたこと

### 4.1 シミュレーションから得られた観察(予備的)

以下は較正完了前の観察であり、確定的な結論ではない。ただし(1)(2)は前述の再現実験を通過した確認済みの所見である。

1. **旧版文書の参照**(確認済み): 新版への改定後も一部のAI従業員が旧版マニュアルを根拠に判断する事象が、独立の再現条件で安定して再現された。
2. **根拠が薄いままの進行**(確認済み): 判断の根拠記録が実質を伴わないまま手続が進行する事象が、同一の内容で再現された。
3. **矛盾規定は読み手の役割によって逆方向に解釈される**: 所管が複数文書で矛盾している規定について、営業側は他部門への引き継ぎを、管理側は自部門での対応を選ぶ傾向が観察された。
4. **未定義の権限用語は寛容な方向に解釈が収斂する**: 禁止事項の曖昧さは全役割が厳格に読む一方、定義のない権限(代理承認者等)は全役割が最も緩い解釈で一致した。未定義の権限が事実上の権限拡大として機能し得ることを示唆する。
5. **統制強化直後の業務停滞**: 期中に証跡要件を強化したところ、システムによる手続拒否が強化前の数倍に増加し、処理件数が大幅に低下した。
6. **既存型モニタリングの盲点**: 現実の組織で用いられる形式のモニタリングルールを再現して適用したところ、証跡不備等は検出する一方、旧版参照や根拠不足などの類型には全く反応しなかった。

### 4.2 較正工程から得られた知見(更新)

較正は「不合格が正しく出ること」「過去の合格を訂正できること」を含めて機能している。主要な経過は以下のとおり。

- **業務再現試験: 合格(再現率87%、目標80%)**。初回は50%で不合格だったが、原因は試験装置がAI従業員に文書検索の手段を与えないまま「文書を調べて回答せよ」と指示していた設計欠陥であった。このときAI従業員の61%が実在しない文書番号を引用しており、根拠の捏造が生じる条件の実例となった。装置修正後、同一の100問で再試験して合格した。引用文書は自己申告ではなく実際の閲覧記録から機械判定する方式に変更済みで、合格した87問は全て実際の文書閲覧に基づく再現である。
- **既知異常の検出試験: 最終確認を実行中**。この試験は較正の統制が最も強く働いた工程であり、次の経緯を経ている。(1) 当初「検出成功」とされた1件は、精査の結果、無関係な定型連絡への単語一致による誤検知であったことが判明し、訂正した。(2) 5種の異常のうち1種は「異常」ではなく統制を改善する良性の介入であり、警報が鳴らないことが正しいと判定基準を再分類した(承認済み)。(3) 異常を仕込まない対照運用でも一部の警報が発火することが判明したため、合否を対照との差分で判定する方式に改めた(承認済み)。(4) 検査シナリオ10種中4種で、設計した状況設定が仮想従業員に届いていない配達不備を発見し修正した。(5) 異常が実際に行動として発現したか(発現しない試行は検出力の証拠にならない)を機械記録する方式を導入した(承認済み)。現在、これら全ての修正を反映した環境で最終の検出試験を実行している。
- **記録の実在性評価: 4回実施、いずれも不合格、ただし毎回敗因を特定し解消**。第1回はシステム内部用語の記録への混入(大差で不合格)、第2回は評価対象の抽出装置の欠陥(同一内容の重複掲載)、第3回は実験条件のラベルが顧客の発言に反映される問題、第4回は社内共有メモの定型の硬直と顧客の発言段階の誤記録であった。各回の指摘は全て原因を特定して修正済みであり、第5回を最終確認試験の後に実施する。なお本評価は現時点ではAIによる代行であり、人間の専門家による評価は対外主張時の必須要件として区別している。
- **判定の較正で残る既知の限界**: 生成AIの言語混濁(まれな不自然表現)は完全には防げない。また、意図的に組み込んだ検証用の状況設定は、多数の記録を並べて読む評価者には設計として認識され得る(これは実験装置である以上避けられない性質であり、評価上は種類を区別して扱う)。

### 4.3 制約事項

- 観察はシミュレーション上のものであり、現実の組織への一般化には検証が必要である。
- 仮想従業員は現時点で単一系統のAIモデルに依存しており、モデル間比較は今後の課題である。
- 記録の実在性評価は現時点でAI代行であり、人間専門家による評価を経ていない。
- 既知異常の検出試験は現時点で各異常1条件の試行であり、対外的に検出性能を主張するには複数条件での統計的補強が必要である。
