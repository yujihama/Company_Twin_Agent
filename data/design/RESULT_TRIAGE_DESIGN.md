# 結果トリアージ設計 — 数百ラン×数千イベントを人間が確認可能にする

前提: fuzzingの成否は生成量ではなくトリアージで決まる。設計原則は3つ。
①LLMにトランスクリプトを全読させない（読むのは新規バケットの代表例だけ）。
②単発イベントを所見にしない（発生率と信頼区間で語る）。
③すべての所見は最小再現バンドルを持つ（再現できない所見はレビュー対象外）。

---

## 1. 一次成果物の形式

全ランはrun bundle（config / world_ledger / attempts / basis_records / chat / meta）を
parquet+JSONLで吐く。分析はDuckDBで束ねて直接クエリ（ETL層を作らない）。
レビュー画面は**LLM→集計→決定論HTMLレンダラ**で静的生成——判断はデータに、描画はコードに。

## 2. オラクル漏斗（3階層のバッチ適用）

```
L0 決定論（全ラン全件・即時）
   ハード制約違反試行(permission-denied含む) / 必須フィールド欠落 /
   KPI 9指標(050) ※高齢者率は分母3定義並列 / 証跡十分性(監査プローブ再構成率) /
   期限超過 / 版数混在 / SoDパターン(申込者=承認者 等)
L1 構造（全ラン・機械）
   承認者集中(座席×例外承認のジニ係数) / 代替承認の連鎖 / 差戻し→再申込の間隔異常 /
   チャット言及→行為の時間相関(黙認シグナル) / 検知漏れ=recorder事実 −(L0∪既存ルール)検出
L2 解釈（新規バケット代表例のみ・LLM）
   L0/L1所見を監査仮説カードへ変換 / novel解釈クラスの命名 / 誤検知の棄却提案
```
L2の入力は常に「バケット代表1–3ラン＋該当span＋basisレコード」であり、生ログ全文ではない。

## 3. 逸脱シグネチャとバケット化（クラッシュ三重化の移植）

```
signature = hash(
  finding_type,                 # 例: evidence_gap / detection_miss / interp_divergence …
  anchor_id,                    # span_id or norm_id or 遷移辺
  role,                         # 座席役割（個人名は除く）
  phase,                        # 説明/適合性/申込/承認/締め
  artifact_skeleton)            # 証跡パターンの正規形（ID・日時・金額をマスク）
```
正規化規則: 固有名・日時・案件IDを型トークンに置換、チャット文面は行為ラベル
（承認依頼/事後報告/相談…）へ写像してからハッシュ。**同型の逸脱が1000ラン分あっても
バケットは1つ**になり、人間は「バケット数×代表例」だけ見ればよい。
バケット属性: 初出config、発生率（config別）、関与spans、S0/S1/S2どこで初出か。

## 4. 最小再現（delta-debugging移植）

新規バケットには自動縮約ジョブを走らせる:
① M1–M4変異を1つずつ外して再現率を測る（効いていない変異を落とす）
② デッキを当該プローブ1枚へ縮約 ③ tickを後方から刈る ④ 座席を関与者のみに
→ 目標形「変異1・プローブ1・座席2–3・5tick前後・再現率付き」のmin-repro bundle。
レビュー担当は数分でリプレイでき、SME盲検にもこの形で出す。縮約で再現率が落ちる場合、
その差分自体が「文脈依存の逸脱」という情報（例: 慣行チケットが無いと出ない＝学習依存）。

## 5. 統計規律

- 指標は**機会あたり発生率**（例: 後追い承認率=後追い件数÷承認機会数）。生カウント禁止。
- config間比較はK seeds上のWilson区間、対アンカーの効果量つき。区間が重なる差は報告しない。
- 解釈分岐はspan×roleの**エントロピー・ヒートマップ**＋クラス分布（registry候補 or novel）。
- 安定度: 同一configのseed間ICC。ICC低のバケットは「モデル気まぐれ」棚へ隔離し、
  Kを増やすか棄却。
- 多重比較: バケットは探索的発見として扱い、昇格（=所見化）には**確認ラン**
  （事前登録した対照ペアでの再現）を必須にする。探索と確認のデータを混ぜない。

## 6. レビュービュー（4画面で足りる）

| 画面 | 内容 | 主な問い |
|---|---|---|
| ① バケット・エクスプローラ | 新規/率変動バケット一覧、min-repro再生リンク | 今週何が新しく起きたか |
| ② 帰属テーブル | Δ=1ペアごとの発生率差と区間 | どの文書変異・ノブが効いたか |
| ③ 分岐ヒートマップ | span×role×解釈クラス、エントロピー、S0→S2の転化率 | どの文言が現場を割るか。解釈は行為になったか |
| ④ 単一ラン・タイムライン | ledger/attempts/basis/chatの時系列合成表示 | この逸脱は世界の中でどう起きたか |

①→④の順でドリルダウンする動線。④は監査調書の下書きを兼ねる（引用span・行為・
証跡・関与者が時系列で揃うため）。

## 7. 運用リズムと監査仮説への出口

- **日次ダイジェスト（自動）**: 新規バケット / アンカー逸脱アラート / 率シフト上位5 /
  昇格待ちconfig。読了5分を上限に設計。
- **週次**: 帰属テーブル更新、確認ラン計画、カバレッジ・プラトー判定、探索方針の変更。
- **所見化**: 「安定バケット（確認ラン再現済み）」のみをL2で監査仮説カード
  （risk_pattern / involved_controls / audit_assertions / evidence_availability /
  existing_rule_detectability / suggested_additional_procedure / limitations）へ変換。
  カードは必ずmin-reproとdivergence行列の該当セルを添付する。
- **検知漏れの定量**: 既存ルール群（閾値・SoD・欠落系）をL0に登録しておき、
  recorder事実のうち既存ルールが沈黙した割合をバケット別に出す——「現行モニタリングの
  弱点マップ」がキャンペーンの副産物として毎週更新される。

## 8. やってはいけないこと

全ランのLLM読解（費用と幻惑の両面で破綻）。バケット未経由の個別事象引用。
seed違いを変異効果として報告。アンカー無しバッチの比較。min-repro無き所見の上申。
