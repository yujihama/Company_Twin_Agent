決定的な副産物: **旧zipが事実上のv1.0コーパス**。改定管理はv1.1行のみを持ち前版を暗示するため、版ずれ条件（D5 stale-document）は合成不要——旧zipの該当文書を特定ロールの索引に残すだけで、完全に自然な二版世界が作れる。

## 2. 決定論境界の更新（Stage 3）

最大の変化は「システム・データ連携仕様メモ」の登場。全45文書が同型5行でシステム挙動を明記しており、SYSTEM判定の文書根拠が初めて得られた:

| 仕様メモの記述 | 判定 |
|---|---|
| 販売イベントID自動採番、未連携の場合は**完了登録を不可**とする | SYSTEM（確定） |
| 承認ステータス: 承認者・承認日時・条件付き承認内容を**必須項目**とする | SYSTEM（フィールド検証） |
| 期限超過時は主管部署と販売品質管理部へ**自動通知**する | SYSTEM（時限イベント。カーネルのスケジューラに追加） |
| 資料版数: 承認済み版**のみ選択できる** ↔ 同じ行の後段「**未承認版が選択された場合は**管理者確認へ差戻す」 | **矛盾（AMB-10として登録）**。選択制限が物理か助言かを仕様自身が決めていない → knob化（K-material-picker） |
| 自由記述のみの場合は要補足**として扱う** | 主語不明（システム自動分類か人か）→ HUMANに倒す |

もう一つ、**050改善アクション4**が「同意ログ、録音ID、資料版数を**申込完了条件に追加する**」（期限8/31、文書管理部）という統制強化計画を文書内に持っている。ラン中日付でK-gateを実際に切り替える「統制変更の効果と副作用」実験（当初構想の検証テーマ7）の台本が、コーパス自身に書かれている。

## 3. 曖昧性採掘v2 — 生存・進化・新規

**生存**: AMB-01（要承認↔例外承認の写像不整合。判断基準5値は37マニュアル全部に残存、条件リストに13バリアント。うち2文書は「例外承認」自体を要承認の条件に挙げる循環）。AMB-02（高齢者定義なし——「歳」の出現**0回**。021の固有論点1が「年齢だけでなく」と未定義を公然と正当化）。AMB-08（048チェックシート12行化後も、行6比較推奨・行7高齢者だけ初期値「該当なし」）。STR-01（コーパス外参照 延べ83・57文書、同名別ID対も残存）。

**進化**: AMB-04→AMB-04'。「後追い登録の場合は理由を記録」条項は削除され、代わりに口頭/チャット承認について**三つ巴の緊張**が生まれた——(a) 044/045判断基準: 口頭承認は「重大」の兆候、(b) 045固有論点3:「後から検索できない状態で残さない」＝検索可能なら残せば可と読める、(c) 全45文書の現場判断事例:「口頭の暫定承認で処理せず、**代替承認者**…をワークフローに記録」。

**新規（v2の目玉）**:
- **CONTRA-01 商品主管機能の帰属矛盾**: 役割と責任表の4行目（「販売対象、説明事項、停止条件、商品改定時の影響を管理する」という同一責任）が、8規程で**7つの異なる部署**に割当てられている（販売統括部×2・チャネル戦略部・デジタル販売管理部・顧客本位推進部・コンプライアンス統括部・販売企画部・販売品質管理部）。一方、部署としての「商品主管部」はdocx側から**消滅**（出現0文書）し、RACIワークブックのR列にだけ生きている。規程間矛盾＋規程vs下位文書矛盾の複合で、v1に欠けていた「本物の文書間矛盾」がこれで揃った。
- **AMB-09 代替承認者**: 全45文書が2回ずつ（計90回）「代替承認者…を記録する」と要求するのに、代替承認者の資格・指名手続・権限範囲の定義が**どこにも無い**。管理者不在プローブの主戦場。
- **AMB-10**: 上記の資料版数ピッカー矛盾。
- **AMB-11 システム別名**: ダミー販売管理システム(8回)とダミー販売管理ワークフロー(77回)、同意ログ基盤と同意管理台帳が併存。旧称混在＝検索・引用の分岐点。

## 4. 世界仕様への反映

カーネル: ステータス機械は維持しつつ、(a) 承認レコードの必須フィールド検証、(b) 期限超過の自動通知（時限イベント）、(c) K-material-picker・K-completion-gate（050改善4の8/31切替イベント）を追加。組織レジスタは049の担当A–Gの7名体制に拡張（加盟店BtoB担当Fと銀行口座担当Gが増え、P-05/routineの座席が自然に埋まる）。エスカレーション先4部署（統括部/品質管理部/コンプラ統括部/危機管理事務局）＋添付証跡要求が045に実体化したので、第二線側の応答レパートリーが文書接地できる。

検証ゲート（Stage 9）への追加: 現場判断事例が文書別になったため、backcasting正例が45×4件に増えた。「管理者不在」事例（全文書共通行）はP-08として新設プローブ化。

## 5. 限界

機械パス偏重のため、文書別実務展開・現場FAQの45×全文は未読（サンプル読解のみ）。norms抽出は境界判定に必要な範囲に限定。多モデル冷読は今回も単一モデル代置であり、役割別解釈予測は帰無仮説として扱うこと。
